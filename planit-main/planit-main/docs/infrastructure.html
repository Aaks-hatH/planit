Cold-start behaviour

PlanIt runs on Render's free-tier infrastructure. The backend web service spins down automatically after 15 minutes of inactivity. When traffic arrives after a spin-down, the service takes approximately 30–60 seconds to cold-start.

During a cold start you may see:

  • A loading spinner or "Service starting up" message on the app homepage
  • A brief error screen on event pages before the API comes back online
  • Delayed Socket.IO connection establishment
Load the check-in page 60 seconds early on event day

Open the check-in dashboard at least 60 seconds before you expect to start admitting guests. This ensures the backend is warm and the guest list cache is populated. If you open it cold at the exact moment the first guest arrives, you may face a brief delay.

Real-time architecture

PlanIt's real-time layer uses Socket.IO over WebSocket with long-polling as a fallback. Every event workspace has its own Socket.IO room, and all clients connected to that room receive updates when any state-changing action occurs.

Redis

Redis serves two roles in PlanIt:

  • Rate limiting state — sliding window counters for all API rate limits are stored in Redis with TTL-based expiry. This ensures limits are consistent even if the Node.js process restarts.
  • Socket.IO adapter — the @socket.io/redis-adapter allows multiple instances of the backend (if scaled horizontally) to share the same Socket.IO room state. Clients connected to different instances still receive the same broadcasts.

Watchdog monitoring

PlanIt runs a watchdog service that pings the backend API at regular intervals and records the results. If the API returns an error or times out, the watchdog logs an incident and the public status page reflects the degraded state within one check interval.

Status page

The public status page at /status shows the live operational state of:

  • API server (response time and last check timestamp)
  • Database (MongoDB connectivity)
  • Redis cache
  • WebSocket server
  • File storage (Cloudinary reachability)
  • Recent incident log (last 30 days)

No account is required to view the status page. Bookmark it if you are running a time-sensitive event.

Why the interface is dark

The dark background is deliberate and functional, not purely aesthetic. Check-in staff typically operate in dim venues — bars, ballrooms, dimly lit entrances. A dark UI dramatically reduces eye strain and makes the critical check-in verdict (green admit / red deny) far more visible at a glance in low-light conditions. The contrast ratio of status colours against the dark background also exceeds WCAG 2.1 AA requirements.

Stars animation

The moving star field visible on the main landing page and event workspaces is a canvas-based CSS animation running entirely on the GPU — no JavaScript animation loop. It uses a repeated CSS animation on SVG circle elements offset by individual delays, which means it is smooth, zero-CPU, and automatically pauses on browsers that honour prefers-reduced-motion.

Page colour system

Each PlanIt page has a distinct accent colour used for interactive elements, highlights, and borders. This makes it immediately clear which interface you're in — organiser workspace (indigo), check-in system (green), guest invite page (gradient), Table Service floor (orange). The colour is applied via a CSS custom property override at the root of each page, so a single design system covers all modes without separate stylesheets.
