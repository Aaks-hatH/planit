Security & Passwords

How PlanIt protects your event data, how the two-password system works, what the anti-fraud engine does, and how guest data is handled.

The two-password system

PlanIt uses two completely separate passwords that serve different purposes. Confusing them is the most common source of access problems.

Account PasswordEvent Password
PurposeProves you are the organiserControls entry to the workspace for everyone
Required?Always (set at event creation)Optional (leave blank for open workspaces)
Who needs it?Organiser only — to claim organiser roleEvery person who opens the event URL
StorageHashed with bcrypt, never readableHashed with bcrypt, never readable
Reset/RecoveryNone — no email on fileOrganiser can change it from Settings

Account passwords cannot be reset

PlanIt stores no email address, so there is no password reset flow. If you lose your account (organiser) password, contact support — but recovery is not guaranteed. Always save your password in a password manager before sharing the event link.

Authentication flow

PlanIt uses a sessionless, JWT-based authentication model. There are no traditional user accounts — identity is tied to the event, not to a registered user profile.

  1. 1
    Organiser creates the event

    Hashed bcrypt credential stored against the event document in MongoDB. The raw password is never persisted.

  2. 2
    Organiser claims role from any device

    Enters the account password on the event page. Backend verifies the bcrypt hash and returns a signed JWT. The JWT is stored in the browser's localStorage for the organiser session.

  3. 3
    Subsequent requests

    Every protected API call includes the JWT in the Authorization header. The server verifies the signature on each request. Tokens are valid for 30 days.

  4. 4
    Staff check-in access

    Staff use a shared PIN provided by the organiser. The PIN is verified against the event's hashed staff credential and returns a scoped JWT with check-in-only permissions.

Response signing

All API responses that return sensitive data — including check-in verdicts — are signed with an HMAC signature. The frontend validates the signature before acting on any check-in result. This prevents a man-in-the-middle attack from injecting a forged "admitted" response if a device is on a compromised network.

Response envelope
{
  "success": true,
  "data": { "admitted": true, "guestName": "Sophie Anderson", ... },
  "sig": "hmac-sha256-of-data-payload",
  "ts": 1718123456789
}

Anti-fraud engine

In Enterprise mode, every check-in attempt passes through a multi-layer anti-fraud middleware before a verdict is returned.

Duplicate detection

At check-in time, the server computes a fingerprint from the guest's invite record and compares it against all fingerprints in the same event. If a match is already marked as admitted, the system flags it as a potential duplicate attempt. When autoBlockDuplicates is enabled in the event security settings, the scan is automatically rejected; otherwise it is shown as a warning for staff to review.

Reentrancy lock

When a QR code is submitted for check-in, the server acquires an exclusive lock on that invite ID for the duration of the validation. If a second device submits the same code while the first is being processed, the second request immediately receives a conflict response. This prevents race conditions at busy multi-lane entrances where two staff members might scan the same code within milliseconds of each other.

Rapid-scan detection

If the same invite code is scanned three or more times within a 10-second window, the system records a rapid-scan event and reduces the invite's trust score. This catches someone methodically testing whether the same QR code works repeatedly.

Trust score

Each invite starts with a trust score of 100. Security events reduce the score:

EventScore impact
Duplicate fingerprint detected−30
Rapid scan (3+ in 10s)−20
Multi-device scan (different IPs, short window)−15
Scan outside time window−10
Manager override used−5 (logged, not a suspicion signal)

Organisers can configure a minimum trust score threshold in the event security settings. Invites below the threshold can be auto-blocked or flagged for manual review.

Time window enforcement

By default, the check-in system only accepts scans within a defined time window:

Default windowValue
Opens before event2 hours
Closes after event start30 minutes

Organisers can extend or disable the window from the check-in settings panel. Scans outside the window receive a clear rejection message explaining when the window opens or that it has closed.

Data privacy & retention

PlanIt collects only the data that event organisers explicitly enter. There are no tracking pixels, no third-party analytics injected into event pages, and no advertising infrastructure.

Deletion is permanent and automatic

The 7-day deletion policy runs daily at 2 AM. Once event data is deleted, it cannot be recovered by PlanIt staff or administrators. Export records you need — guest lists, attendance logs, uploaded files — before the deletion window closes.

How guest email addresses are used

Guest email addresses collected during invite creation are used exclusively for duplicate fingerprinting — the system uses them to detect if the same person is invited twice under different names. PlanIt does not send emails to guests automatically, display email addresses to guests, or use them for any marketing purpose. Emails are stored hashed in the duplicate detection index alongside the full email for organiser reference.

The no-account model

PlanIt deliberately avoids collecting user accounts for guests and participants. This design decision has several security and privacy benefits:

  • No password database to breach for guests (there are none)
  • No persistent user profiles that accumulate data across events
  • No email address required from guests — zero friction for attendance
  • No risk of guest credentials being reused or phished
  • Identity data (guest names, emails) is tied to a specific event and deleted with it

The trade-off is that sessions are ephemeral — if a participant closes their browser, they need to re-enter their display name when they return. There is no "log back in" flow for participants because there is nothing to log back into.
