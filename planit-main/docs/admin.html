Admin Panel

System-wide oversight for PlanIt platform administrators — events, users, analytics, marketing, system health, security, logs, and the command centre.

Platform administrators only

The admin panel at /admin is for PlanIt platform administrators, not event organisers. If you are an event organiser, you do not have access to the admin panel and do not need this section.

Overview

The admin panel provides a global view of all events running on the PlanIt platform. It is accessed at /admin and requires a separate admin credential distinct from any organiser account. The panel is a React SPA with a sidebar navigation and 15 distinct sections.

Events management

The events table shows every active and recent event on the platform with:

  • Event name, slug, and mode (Standard / Enterprise / Table Service)
  • Organiser identity and join date
  • Participant count and guest count (Enterprise)
  • Check-in rate and last activity timestamp
  • Days until deletion countdown (non-Table Service events)

Administrators can search by event name or slug, filter by mode or date range, view full event details and guest lists, and in exceptional cases force-close or immediately delete events.

Analytics

The analytics section provides platform-wide aggregate metrics including:

  • Total events created per day/week/month (chart)
  • Total invites sent and admission rates across all events
  • Mode distribution: what fraction of events use Standard vs Enterprise vs Table Service
  • Most active features: which workspace tools see the most engagement
  • Peak usage times: which days and hours see the most check-in activity
  • Returning organiser rate: what fraction of organisers have created more than one event

Analytics are anonymised at the individual guest level — no personally identifiable information is surfaced in aggregated views.

The security section exposes platform-level security controls:

IP bans and session blocks

Administrators can view all active IP bans and session blocks, see which security events triggered them, and manually unblock any entry. Bans are stored in Redis with TTL equal to SECURITY_BAN_MINUTES.

Marketing tools

The marketing section provides tools for communicating with the platform's organiser base.

  • Platform announcements — publish notices visible to all organisers when they log in. Used for downtime warnings, feature releases, policy changes.
  • Email campaigns — send targeted emails via the configured transactional email provider (Brevo or Mailjet) to organiser segments. All emails comply with applicable anti-spam laws and operate only on contacts who have used PlanIt.
  • Waitlist management — view, approve, or reject entries on the platform access waitlist (if waitlist mode is active).

System health

The system view shows real-time health for all platform components:

  • API server response times and error rate over the last hour
  • MongoDB connection pool status and query latency
  • Redis connection status and memory usage
  • Socket.IO WebSocket server connection count and throughput
  • Cloudinary storage usage and API quota remaining
  • Uptime history and incident log for the past 30 days

Logs

The logs section provides access to:

  • Server logs — Node.js application logs with level filtering (info, warn, error)
  • Access logs — HTTP request log with endpoint, method, response code, latency, and IP
  • Audit log — every admin action with who performed it, what was changed, and the before/after state
  • Security event log — all anti-fraud triggers, IP bans, failed auth attempts, and manual overrides

Data retention management

PlanIt's retention policy deletes all event data seven days after the event date. The deletion job runs automatically at 2 AM daily. From the admin panel, administrators can:

  • View which events are scheduled for deletion in the next 24 hours
  • See a full list of events deleted in the last 30 days
  • In exceptional circumstances, manually trigger or defer deletion of a specific event
Deletion is permanent — for everyone

Once event data is deleted by the retention policy or manually, it cannot be recovered by admins, organisers, or support staff. There are no backups that restore individual event data. This is by design to protect guest privacy.

Bug reports & support tickets

User-submitted reports from /support are routed to the admin panel. Each report shows:

  • Submitted timestamp and user browser/OS info (auto-collected)
  • User-written description of the issue
  • Category (bug, feature request, account issue, other)
  • Status: Open, In Progress, Resolved, Closed

Administrators can change status, add internal notes, and mark reports as resolved.

Command Centre

The Command Centre provides direct server operations for platform maintenance:

  • Run retention cleanup now — immediately execute the deletion job rather than waiting for the 2 AM cron
  • Flush Redis cache — clear all rate-limit counters and session caches (use with caution — this resets all active rate windows)
  • Restart WebSocket server — gracefully restart the Socket.IO layer and re-connect all clients
  • Force reconnect watchdog — re-initialize the watchdog monitoring ping loop
  • Run database diagnostics — execute a suite of read/write tests and report latency
